1. 核心思路

上下文压缩 = 节点折叠 + 存磁盘留引用 + 时间降级。所有操作都作用在 DAG 节点上。

2. 节点生命周期管理

我们独有的设计。Claude Code 是线性对话没有 DAG，不需要这一层。

2.1 函数运行中

触发
🟢 函数节点 被调用，status="running"
做什么
内部子树（🟡 工具调用、嵌套 🟢 函数、LLM 交互）完整展开在上下文中
可见范围
由 render_context 的 callers / subcalls / expose 控制
节点状态
函数内部视角（函数自己的 LLM 调用）：子节点 full（完整渲染 input + output）
外部视角（调用方看这个函数）：默认 expose=io，调用方只看到输入和输出，看不到内部的 🟡 工具调用 和 LLM 交互
涉及节点
🟢 函数节点（ROLE_CODE）+ 其下所有 🟡 工具子节点（仅在函数内部视角参与渲染）
函数调用特殊设计
render_context 的 frame_entry_seq 参数划分"函数前历史"和"函数内历史"，callers 控制能看多少父辈。expose=io 确保内部细节不泄漏给调用方

2.2 函数结束 → 自动折叠

触发
🟢 函数节点 返回，status 从 "running" 变为 "completed" / "error"
做什么
a. 内部子树（所有子节点）序列化存磁盘
b. 存储路径：~/.openprogram/sessions/<sid>/collapsed/<node_id>.jsonl
c. 节点变为折叠状态，只保留一行（输入→输出）：
research_agent("LLM reasoning") → 5 ideas (详见 collapsed/abc123)
这和默认 expose=io 天然一致——外部本来就只看到输入输出
d. → 折叠后的节点进入 §3-§5 的常规压缩流程，和 🔵 对话节点 一视同仁
各场景
大模型调用的函数：自动折叠
手动调用的函数：同样折叠，只保留输入输出
嵌套函数：内层先折叠，外层结束时再折叠
涉及节点
🟢 函数节点 从 status=running 变为 completed，子节点从渲染列表移除（不物理删除）
函数调用特殊设计
这是 🟢 函数节点 独有的步骤，🔵 对话节点 不经过这步

2.3 存磁盘的格式和检索

格式
JSONL，每行一个子节点（role / name / input / output / metadata）
检索
大模型可以通过 read_file 工具读取路径，查看函数运行的具体过程
关联
→ §4.1 Microcompact 也存磁盘，共用同一套存储目录结构
涉及节点
存的是被折叠的子节点，父节点（🟢 函数节点 本身）留在 DAG 中
3. 每轮常规操作

每次 LLM 调用前都做，和上下文长度无关。

3.1 截断超大单个输出

触发
每次 LLM 调用前检查所有节点的 output
做什么
单个工具输出超过阈值（比如 10000 tokens）就截断。保留开头 N tokens + 结尾 M tokens，中间用 [... truncated ...] 替代
作用对象
所有节点：🔵 对话 🟢 函数 🟡 工具（含 §2.2 折叠后的函数节点）
涉及节点
作用在单个节点的 output 字段上，不改变节点状态，不影响节点间结构
函数调用特殊设计
无。§2.2 折叠后的 🟢 函数节点 只有一行，通常不会触发截断
4. 空闲时清理

用户一段时间没发消息时做。

4.1 旧工具输出存磁盘（Microcompact）

触发
距上次用户交互超过阈值（比如 30 秒）
做什么
按时间排序，最旧的 🟡 工具输出 存磁盘，上下文中替换为引用路径
存储
~/.openprogram/sessions/<sid>/tool_outputs/<node_id>
上下文中变成
[output stored: tool_outputs/abc123, retrievable by read_file]
保留条件
最近的工具输出保持 inline（不清理）。大模型需要时可以通过 read_file 读回来
关联
→ 和 §2.3 共用存储机制，但针对的是单个 🟡 工具输出 而非 🟢 函数子树
涉及节点
修改 🟡 工具节点 的 output 字段（替换为引用路径），节点本身保留在 DAG 中
函数调用特殊设计
🟢 函数运行中（§2.1）：内部的 🟡 工具输出 也可以被清理（运行时间长时旧的内部输出存磁盘）
🟢 函数结束后（§2.2）：内部子树已整体存磁盘，Microcompact 不再作用于这些子节点
5. 阈值触发的压缩

上下文占比超过阈值时做。

5.1 触发条件和目标

自动触发
history tokens 超过 context_window - 13K（约 83.5% 占用），和 Claude Code 一致
建议手动触发
用户在 60% 时手动 /compact，质量比自动好
目标
不设固定百分比，逐级压缩（Snip → 摘要）直到能继续调用 LLM
检查时机
每次 LLM 调用前（§3 之后）
涉及节点
计算所有参与渲染的节点的 token 总和

5.2 按时间删旧节点（Snip）

触发
§5.1 条件满足
做什么
直接删除最旧的节点（不做摘要，从渲染列表移除）
作用对象
所有类型：🔵 对话 🟢 已折叠函数 🟡 工具
删多少
从最旧的开始删，直到总 token 数降到目标以下
涉及节点
设置节点 visibility=hidden，render_context 跳过这些节点。数据仍在 DAG 中（不物理删除）
不够时
→ 进入 §5.3
函数调用特殊设计
§2.2 折叠后的 🟢 函数节点 被当作普通节点，按时间排序参与 Snip
正在运行的 🟢 函数（§2.1）不参与 Snip（不能删正在用的）

5.3 对话节点 LLM 摘要

触发
§5.2 Snip 后仍超目标
做什么
用 LLM 把旧 🔵 对话节点 摘要成一段文本
摘要预生成
每轮 turn 结束时异步生成 1-2 句摘要，存在节点 metadata.summary 中
压缩时
直接用缓存的 summary 替换完整对话，不需要额外 LLM 调用。如果没有预生成的 summary 则调 LLM（更慢，但保底）
摘要后变成
[turn 3] 用户问了代码性能 → 模型发现3个瓶颈（约 50 tokens）
关联
→ §2.2 折叠后的 🟢 函数节点 已经是一行，通常不再摘要；但如果太长也可进一步缩短
涉及节点
节点 visibility 从 full 变为 degraded，渲染时用 summary 替代完整内容
函数调用特殊设计
🔵 对话节点：用预生成的摘要
§2.2 折叠后的 🟢 函数节点：已是一行，通常不再摘要
正在运行的 🟢 函数（§2.1）：不摘要（正在用）

5.4 /compact 手动命令

触发
用户输入 /compact 或 /compact <提示词>
做什么
立即触发 LLM 全量摘要（不经过 §5.2 Snip）。可以带提示词指导保留什么
示例
/compact 保留数据库迁移的讨论
摘要后
所有旧节点 visibility=hidden，生成一个新的 compaction summary 节点。只保留最近 1-2 轮完整
涉及节点
所有旧节点（🔵 对话 + 🟢 已折叠函数）一起被摘要
函数调用特殊设计
无。已折叠的 🟢 函数节点 和 🔵 对话节点 一起被摘要
6. 完整流程推演

场景：用户在 200K context 下做编程任务，中间调用了 research_agent 函数。

阶段 1（0-30%）：正常对话
🔵 user → 🔵 assistant → 🔵 user → 🔵 assistant
没有任何压缩触发。所有 🔵 对话节点 visibility=full。
无章节触发
阶段 2（30-50%）：调用函数
🔵 user: "帮我调研 LLM reasoning"
🔵 assistant 调用 🟢 research_agent（§2.1 生效）
🟢 research_agent 运行中：内部 15 轮 🟡 工具调用 展开在上下文
某个 🟡 search_papers 返回 20000 tokens → 截断到 10000
§2.1 + §3.1 + §4.1
阶段 3：函数结束
🟢 research_agent 返回 5 个 idea
§2.2 触发：内部 15 轮子树存磁盘（collapsed/abc123.jsonl），节点折叠成一行
上下文释放约 30K tokens
折叠后的节点和 🔵 对话节点 一样参与后续压缩
§2.2
阶段 4（50-80%）：继续对话
用户继续讨论、改代码
§4.1：空闲时旧的 🟡 read_file 输出存磁盘
turn 结束时预生成 🔵 对话 摘要（§5.3 的准备）
§4.1 + §5.3 预生成
阶段 5（~83.5%）：触发自动压缩
§5.1：超过 context_window - 13K（约 83.5%）自动触发
§5.2 Snip：删最旧的 10 轮节点（含已折叠的 🟢 research_agent）
如果还不够 → §5.3：用预生成的 summary 替换更多旧 🔵 对话
逐级压缩直到能继续调用 LLM
§5.1 → §5.2 → §5.3
阶段 6：继续使用
压缩后正常对话。再次到 ~83.5% 时重复阶段 5（链式压缩）
大模型想回顾 🟢 research_agent 的具体过程：read_file("collapsed/abc123.jsonl")
§5 循环 + §2.3 检索

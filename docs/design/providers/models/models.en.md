# Model Catalog and Provider Configuration

> The control logic for thinking effort is covered in [thinking-effort.md](thinking-effort.md). This document covers the data layout of the model catalog, the configuration structure, the Fetch flow, and data merging.

## 1. Core Principles

**Every provider is self-contained.** All configuration (thinking mappings, probe scripts, Fetch data) lives under `providers/<provider>/`. Adding a new provider = adding a folder; adding nothing also works, falling back to OpenAI compatibility.

The two kinds of data are stored separately:

| | thinking.json | models.json |
|---|---|---|
| Content | API parameter format, effort mapping, per-model override | The model list pulled by Fetch (id, context, reasoning, etc.) |
| Change frequency | Once every few months (only when the API format changes) | Overwritten on every Fetch |
| Version control | In git | .gitignore |

## 2. File Layout

```
openprogram/providers/
├── anthropic/
│   ├── anthropic.py             ← stream implementation
│   ├── thinking.json            ← in git: thinking effort mapping
│   ├── probe_thinking.py        ← probe script run automatically during Fetch
│   └── models.json              ← .gitignore: model list generated by Fetch
├── deepseek/
│   ├── thinking.json
│   ├── probe_thinking.py
│   └── models.json
├── google/
│   ├── google.py
│   ├── thinking.json
│   ├── probe_thinking.py
│   └── models.json
├── openai_codex/
│   ├── openai_codex.py
│   ├── thinking.json
│   ├── probe_thinking.py
│   └── models.json
├── github_copilot/
│   ├── thinking.json            ← wire_format: "none" (thinking not supported)
│   └── ...
├── thinking_spec.py             ← shared: load thinking.json + translate + derive
├── thinking_catalog.py          ← shared: populate the thinking fields of Model objects at startup
└── _shared/                     ← shared utility functions
```

`.gitignore` has a line `openprogram/providers/*/models.json`, so the models.json of every provider stays out of git.

### 2.1 Providers Without a Folder

Community providers (groq, mistral, etc.) can have no folder. In that case:
- **thinking.json**: `thinking_spec.get_thinking_spec()` returns the OpenAI-compatible fallback (`effort_string` + `low/medium/high`)
- **probe_thinking.py**: during Fetch, `_load_probe()` catches ImportError and silently skips
- **models.json**: during Fetch, `_provider_dir()` automatically creates the folder and writes to it

### 2.2 Provider Alias

`claude-code` and `anthropic` use the same API and models. `thinking_spec._THINKING_ALIASES = {"claude-code": "anthropic"}` lets claude-code share anthropic's thinking.json without copying the file.

## 3. thinking.json Structure

Declares how this provider's API receives the thinking/effort parameter.

### 3.1 Field Definitions

| Field | Type | Description |
|---|---|---|
| `wire_format` | `"effort_string"` / `"budget_tokens"` / `"none"` | Whether the API accepts a string, a token count, or does not support thinking |
| `effort_map` | `{framework level: API string}` | The mapping table when `wire_format="effort_string"` |
| `budget_map` | `{framework level: token count}` | The mapping table when `wire_format="budget_tokens"` |
| `default_effort` | `string` / `null` | The default level when the user makes no selection |
| `model_overrides` | `{model_id: {...}}` | Override configuration for specific models |
| `model_overrides.<id>.effort_map` | empty dict `{}` | Indicates the model does not support effort adjustment |

### 3.2 Examples

**Anthropic** — effort string, with model_overrides written automatically from API capabilities:

```json
{
  "wire_format": "effort_string",
  "effort_map": {
    "minimal": "low", "low": "low", "medium": "medium",
    "high": "high", "xhigh": "xhigh", "max": "max"
  },
  "default_effort": "high",
  "model_overrides": {
    "claude-opus-4-8": {
      "effort_map": {"low":"low","medium":"medium","high":"high","xhigh":"xhigh","max":"max"}
    },
    "claude-opus-4-5-20251101": {
      "effort_map": {"low":"low","medium":"medium","high":"high"}
    }
  }
}
```

The override for Opus 4.8 has only 5 keys (no minimal), so the UI shows 5 levels. Opus 4.5 has only 3 keys, so the UI shows 3 levels. The provider-level `effort_map` has 6 keys, serving as the fallback for models without an override.

**DeepSeek** — V4 has five levels, R1 has no effort control:

```json
{
  "wire_format": "effort_string",
  "effort_map": {
    "minimal": "minimal", "low": "low", "medium": "medium",
    "high": "high", "max": "max"
  },
  "default_effort": "medium",
  "model_overrides": {
    "deepseek-reasoner": {"effort_map": {}},
    "deepseek-chat": {"effort_map": {}}
  }
}
```

The `effort_map` of `deepseek-reasoner` is an empty dict `{}` — meaning this model has reasoning capability but does not support effort adjustment (R1 always reasons at full strength). The empty dict for `deepseek-chat` means V3 does not support reasoning.

**Google Gemini** — uses a token count instead of a string:

```json
{
  "wire_format": "budget_tokens",
  "budget_map": {
    "minimal": 512, "low": 2048, "medium": 8192,
    "high": 24576, "xhigh": 32768, "max": 65536
  },
  "default_effort": "medium"
}
```

**GitHub Copilot** — does not support thinking:

```json
{
  "wire_format": "none",
  "default_effort": null
}
```

### 3.3 Derivation Rules for thinking_levels

How many levels the UI shows is not maintained manually; it is derived automatically from the keys of the mapping table:

1. `reasoning=false` → empty list, the UI does not show the slider
2. Has `model_overrides` and `effort_map` is not null → use the override's keys (empty `{}` → empty list → the UI does not show it)
3. No override → use the keys of the provider-level `effort_map`/`budget_map`
4. No thinking.json → the fallback keys (low/medium/high)

## 4. models.json Structure

Generated by Fetch Models, stored in the provider folder, not in git:

```json
{
  "provider": "deepseek",
  "models": [
    {
      "id": "deepseek-v4-flash",
      "name": "deepseek-v4-flash",
      "reasoning": true,
      "thinking_levels": ["minimal", "low", "medium", "high", "max"],
      "default_thinking_level": "medium",
      "context_window": 1000000,
      "max_tokens": 384000,
      "input_cost": 0.14,
      "output_cost": 0.28
    }
  ]
}
```

Anthropic's models.json also includes a `supports_adaptive` field (obtained from the capabilities API).

## 5. Fetch Flow

Triggered when the user clicks "Fetch Models" in Settings:

```
1. fetchers/__init__.py: fetch_models_remote(provider_id)
       │
       ▼
2. select fetcher (the _FETCHERS table or the generic _fetch_openai_compat)
       │
       ▼
3. call the provider's API (e.g. /v1/models) to get the model list
       │
       ├── Anthropic: additionally call GET /v1/models/{id} for each model
       │   → _extract_thinking_caps() extracts effort/adaptive capabilities
       │
       ├── others: _load_probe(provider_id) calls probe_thinking.probe()
       │   → infer reasoning capability from the model id
       │
       ▼
4. enrichment: merge models.dev data (pricing, reasoning flag)
       │
       ▼
5. write to providers/<provider_dir>/models.json
       │
       ▼
6. frontend re-requests /api/agent_settings → UI refreshes
```

## 6. Data Merging: Three-Layer Stacking

`list_models_for_provider()` in `listing.py` is the single exit point for model data. It merges three layers of data into the final result:

```
Layer 1: combined_models(provider_id)
         = models.json (Fetch data) + models.dev (pricing/caps)
         → produces the base info for each model (id, reasoning, context_window, ...)

Layer 2: thinking_spec.derive_thinking_fields()
         = read thinking.json → model_overrides → provider-level mapping → fallback
         → produces thinking_levels / default / variant

Layer 3: the thinking_levels in the Fetch data (used when Layer 2 yields no result)
         → thinking_levels provided by models.dev or API capabilities
```

The merged result is consumed by three consumers:

| Consumer | Purpose |
|---|---|
| `list_models_for_provider()` | The model table on the Settings page |
| `list_enabled_models()` | The model selector on the Chat page (delegates to the above) |
| `get_thinking_config_for_model()` | The options list of the UI thinking picker (delegates to the above) |

All three consumers go through the same path, ensuring the frontend display and backend data are fully consistent.

## 7. The Role of models.dev

`models.dev` is a cross-provider general data source, providing:
- The model list for providers that have never been fetched (as a backstop)
- pricing (official APIs usually do not return prices)
- reasoning flags, capability details

Cached in memory, with a 24h TTL, lazily refreshed at startup. It is not stored in a provider folder (it is cross-provider and belongs to no single one).

## 8. Steps to Add a New Provider

### 8.1 Providers With a Dedicated API (e.g. DeepSeek)

1. Create the `providers/<name>/` directory
2. Write `thinking.json` (declare wire_format, effort_map, and any necessary model_overrides)
3. Write `probe_thinking.py` (expose a `probe()` function that infers reasoning from the API or the model id)
4. Add a dedicated fetcher in `fetchers/` (or use the generic `_fetch_openai_compat`)
5. The user clicks Fetch → models.json is generated automatically → the UI displays it automatically

### 8.2 Community Providers (e.g. groq, mistral)

Nothing needs to be done. The user fills in the API key in Settings and clicks Fetch:
- Fetch uses the generic `_fetch_openai_compat`
- thinking.json missing → the fallback is used automatically (low/medium/high)
- probe_thinking.py missing → silently skipped
- the folder is created automatically when models.json is written

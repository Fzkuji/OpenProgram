OVERVIEW

事件层是什么

一条进程级单例总线（openprogram/agent/event_bus.py）。框架各处的「源」把事件 emit 进总线；「消费者」按事件类型 subscribe。源和消费者互不认识，只认总线——这就是「统一」。事件分两大类：

A 类 · agent 活动：agent 干活过程中发生的（模型回复、工具前后、文件改、turn 结束…），通常带 session/turn 关联。
B 类 · 系统状态：全局状态变化（凭据限流、上下文压缩、外部消息、技能变更…），可能不属于任何 turn。
ws.frame：透传信封——外部源把现成的前端 WS 帧经总线送往浏览器，webui 订阅后原样广播。

配套设计文档：event-layer.html（模型与定位）· framework-evolution.html（框架演进）。

DATA MODEL

Event 数据模型

每条事件都是一个 frozen dataclass。核心三样（是什么事 + 内容 + 时间）固定，关联信息放进开放的 metadata 口袋。openprogram/agent/event_bus.py:39-49

@dataclass(frozen=True) class Event: id: str # 唯一编号 ts: float # 发生时间（unix 时间戳） type: str # 事件类型，见下方目录 origin: str # 谁引起：user/agent/tool/system/proactive payload: dict # 这件事的内容（每类事件不同） metadata: dict # 开放口袋：{"session":…, "turn":…}，需要才有
不可变（frozen）= 流水账只记不改，多线程共享天然安全。metadata 由 make_event 自动从 ContextVar 填上 session/turn（agent 活动事件有，系统事件通常没有）。

API · CORE

总线 API

get_event_bus()
def get_event_bus() -> EventBus
取进程级单例总线（双检锁，照 AuthStore/TaskRunner 先例）。同一 worker 进程任意线程拿到同一个实例。所有 emit/subscribe 都从它来。
openprogram/agent/event_bus.py:241-251
EventBus.subscribe(handler, *, types=None)
def subscribe( self, handler: Callable[[Event], Any], *, types: set[str] | None = None, ) -> Callable[[], None]
注册一个类型化订阅者。types=None 收所有事件；types={"tool.before","file.changed"} 只收这几类。返回一个 unsubscribe() 函数。handler 在 emit 所在线程被调用（同步函数；async handler 在无 loop 的 worker 线程会被静默跳过）。
openprogram/agent/event_bus.py:159-178
EventBus.emit(event)
def emit(self, event: Event) -> None
把一条 Event 分发给所有匹配的订阅者。fire-and-forget——某个 handler 抛异常不影响其他 handler、不影响 emit 调用方。typed 事件路径与 legacy on(channel) 路径互不串。

API · HELPERS

发事件的 helper（源用这些）

emit_safe(type, origin, payload=None, metadata=None)
def emit_safe( type: str, origin: str, payload: dict | None = None, metadata: dict | None = None, ) -> None
源发事件的便捷入口：构造 Event（自动填 id/ts + 从 ContextVar 补 session/turn）并 emit，吞掉一切异常——事件层绝不能弄崩发它的业务代码。框架里所有 A/B 类事件都用它发。
openprogram/agent/event_bus.py:96-107
emit_ws_frame(frame)
def emit_ws_frame(frame: dict) -> None
外部源用：把一个现成的前端 WS 帧（{"type":…, "data":…}）装进 ws.frame 事件经总线送往前端。让 task runner / channels / worktree 等不再 import webui。同样吞掉一切异常。
openprogram/agent/event_bus.py:110-124
make_event(type, origin, payload=None, metadata=None)
def make_event(...) -> Event
底层构造器：生成带 id/ts、自动补 ContextVar 关联的 Event（不 emit）。需要拿到 Event 对象自己处理时用（如 tool.before 既要 emit 又要喂给 gate）。emit_safe 内部就是 emit(make_event(...))。

GUIDE ①

观察：订阅事件做只读响应

最常见的用法——盯某几类事件，发生时做点只读的事（统计、日志、转发、触发后台活）。异步、不挡 agent。

from openprogram.agent.event_bus import get_event_bus def on_event(ev): if ev.type == "file.changed": print("改了", ev.payload["path"], "in session", ev.metadata.get("session")) # 只收这两类；返回 unsubscribe() unsub = get_event_bus().subscribe(on_event, types={"file.changed", "tool.after"}) # ... 用完 unsub() 取消
防自触发：如果你的订阅者会再 emit 事件，注意别和自己产生的事件绕成环——检查 ev.origin == "proactive" 或因果根来短路（见附录·不变式）。
GUIDE ②

拦截：在工具执行前拦下来

全框架唯一的同步拦截点，只在 tool.before。注册一个 gate 函数：拿到 tool.before 事件，返回 None 放行 或 一个 deny 理由字符串拦下。deny 理由作为 error tool result 回给模型。多 gate 取最严。对 subagent 同样生效（绕过 permission_mode=bypass）。

from openprogram.agent.tool_gate import register_tool_gate def guard(ev): args = ev.payload.get("args") or {} if ev.payload.get("tool") == "bash" and "rm -rf /" in str(args.get("command", "")): return "危险删除，拦下" # 非 None = 拦截 return None # 放行 unregister = register_tool_gate(guard)
gate 必须快：它挡在工具执行的同步路径上，agent 在等它给准话。不要在 gate 里调 LLM / 读网络 / 做慢活。gate 自身抛异常按放行处理（fail-open）。openprogram/agent/tool_gate.py
GUIDE ③

发问：函数中途停下来问用户

在任意函数体里调 runtime.ask / runtime.confirm，阻塞到用户在前端回答。底层发出 question.asked 事件（前端弹出问题卡片），用户答后函数 resume。三态显式。

# 单选 / 多选 / 自由文本 answer = runtime.ask( "用哪个日期库？", options=["dayjs", "luxon"], # 可选；None = 纯自由文本 multi=False, # True 返回 list[str] allow_custom=True, # 允许选项外自由输入 timeout=300, # 秒，默认 5 分钟 default=None, # 超时返回它；无 default 超时抛 AskTimeout ) # 用户点「拒绝」→ 抛 UserDeclined # 是 / 否 ok = runtime.confirm("要归档全部 87 封邮件吗？", default=False) # 超时返 default，不抛 # headless（无前端）跑时分支 if runtime.can_ask(): name = runtime.ask("项目名？")
三态：答了→返回答案；用户拒绝→抛 UserDeclined；超时→confirm 返 default、ask 有 default 返 default 否则抛 AskTimeout。替代了旧的「300s 静默返回 None」。openprogram/agentic_programming/runtime.py（ask/confirm/can_ask）· openprogram/agent/questions.py（QuestionRegistry）
当前范围：同进程的 @function 工具可用；@agentic_function 跑在子进程，需要 parent→child 答案通道（Phase 2，尚未做）。
INTEGRATION

桥接：把已有信号引进总线

auth 子系统自带规范的 subscribe/_emit，所以走「真桥」：订阅它、翻译成统一 Event 再 emit。worker 启动时装一次。openprogram/agent/event_bridges.py · 装于 openprogram/worker/runner.py:263-267

AuthEventType→ 总线事件
POOL_MEMBER_COOLDOWNcredential.cooldown
POOL_ROTATEDcredential.rotated
POOL_EXHAUSTEDcredential.exhausted
只桥这三种「有响应价值」的（限流/轮换/耗尽），登录/刷新/档案生命周期不进总线——遵循「有消费者想响应才是事件」的边界原则，避免事件流变垃圾场。context / channels / memory / skills 是自家代码，直接源头 emit_safe，不经过桥。
INTEGRATION

前端消费

webui 是总线的一个订阅者（启动钩子 _subscribe_event_bus，openprogram/webui/server.py:1166-1189）：订阅 ws.frame 事件，把里面的原始帧 _broadcast 给所有 WebSocket 客户端。前端 web/lib/use-ws.ts 按帧 type 分发处理——所以「事件 → 浏览器」这条路是通的。前端认识的帧包括 task_status、channel_turn、agent_session_updated、worktree_status、programs:changed、skills:changed、question.asked/replied/rejected 等（见上方 ws.frame 目录）。

INTEGRATION

调试：把事件流打出来看

设环境变量 OPENPROGRAM_EVENT_LOG，每条事件作为一行 JSON 追加落盘。1/true → /tmp/openprogram-events.jsonl，其他值当作路径。openprogram/agent/event_bus.py:256-285

# 启用事件日志后重启 worker，发一条消息，看完整事件序列 OPENPROGRAM_EVENT_LOG=1 openprogram worker restart # 然后 cat /tmp/openprogram-events.jsonl
一个普通 chat turn 会依序打出 user.prompt_submitted → model.response_started → tool.before → tool.after → model.response_completed → turn.ended，每条带 metadata 里的 session/turn。这是确认事件层在跑的最快方式。

附录

origin 枚举（事件由谁引起）：

user用户引起（发消息、外部通道来消息、答问题）
agent模型/主循环引起（模型回复、工具调用、turn）
tool工具引起（工具完成、文件改动）
system系统状态引起（凭据、上下文、记忆、技能、子任务）
proactive框架主动性自己产生（保留——proactive 策略未激活）

防自触发不变式：origin=proactive 的事件不应再触发会产生新事件的动作，斩断自激环。当前 proactive 策略层未激活，但这条底线在事件层设计里成立。

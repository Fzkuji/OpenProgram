# 未来工作 —— 明确不做的部分

这些是在设计讨论中考虑过、但我们**选择不做**的条目。在此记录，是为了让下一个想到同样点子的人能找到此前的推理，不必从头重做分析。

我们在这次设计讨论进行到一半时采纳的规则：**默认与参考实现保持一致（claude-code / opencode / hermes）；只有在存在具体的用户痛点时才新增机制**。下列条目在被提出的当时都没通过这条测试。

如果后续有用户把其中某一项当作真实问题碰到了，应当用其症状更新本文档，并推进实现。

---

## 1. 按会话的 gating 开关

**点子**：让用户为某一次聊天会话覆盖 agent profile —— “在这次对话里，也启用 `prod-deploy`”。

**提出的理由**：偶尔某个 agent 角色的 gating 对于一次性任务来说过于收紧，而编辑 profile 又显得太重。

**我们没做的理由**：
- 没有任何参考框架有这个。claude-code 的 `permissionMode` 是按 agent 的，不是按会话的。opencode 的 ruleset 是按 agent 的。hermes 没有按会话的逃生口。
- 现有工作流是：编辑 `agent.json`，保存，下一轮就会读取生效。热重载很快。
- 加上按会话的覆盖会带来一个状态管理问题（覆盖存在哪里？刷新页面后还在吗？会在 WS 重连之间泄漏吗？），在没有证据表明更简单的“编辑 profile”工作流不够用之前，我们不想去解这个问题。

**何时重新考虑**：如果用户开始提类似“我为了这一个调试任务反复来回编辑 agent.json”的 issue。

---

## 2. 插件子进程沙箱

**点子**：在子进程里运行插件代码，限制其文件系统 / 网络访问 —— 用 `subprocess.Popen` 配合 seccomp-bpf 或类似手段。

**提出的理由**：插件带的是任意 Python 代码；恶意插件可能外泄 API key 或 rm -rf 整个工作区。

**我们没做的理由**：
- claude-code、opencode、hermes 都是在进程内加载插件的，没有一个对插件代码做沙箱。
- 我们已经有插件**信任级别**（`plugin.json` 声明 `trust: "verified" | "community"`），并且我们是在信任级别而非操作系统级别上对危险能力做 gating。
- 在 macOS / Windows 上做真正的沙箱确实很难 —— seccomp 只限 Linux，App Sandbox 只限 macOS，二者都覆盖不了跨平台开发的场景。
- 威胁模型还不足以justify这份工程成本：插件安装是一个带 manifest 审查环节的显式用户动作。

**何时重新考虑**：如果我们哪天支持**自动**插件安装（例如让 LLM 在对话中途自行决定安装某个插件），威胁模型就变了，沙箱就值这份成本了。

## 3. Skill 的 `requires` 链

**点子**：在 SKILL.md frontmatter 里加一个 `requires: [other-skill]` 字段 —— 调用 `prod-deploy` 时自动先加载 `kubectl-helpers`。

**提出的理由**：skill 有时依赖其他 skill 的上下文。如今用户得手动串起 `/skill A /skill B`。

**我们没做的理由**：
- 任何参考框架里都没有。Anthropic 的 SKILL.md 规范没有 `requires` 字段。
- 由 LLM 中介的选择模型假定模型会读取所有 SKILL 描述并自行挑选。如果 skill A “需要” skill B，正确答案通常是“把它们合并”或“在 A 的描述里写明它和 B 一起用效果最好”—— 而不是新增一张需要用户维护的依赖图。
- 依赖解析会引入排序问题（依赖在父项之前加载吗？）、冲突问题（两个 skill 各自需要互不兼容的第三方 skill？），以及版本问题，而这些问题目前在代码库的其他任何地方都还不存在。

**何时重新考虑**：如果 skill 作者开始反复以人类可读的形式写下“把这个 skill 和 X 一起用”，那我们就有了真实信号，说明 `requires` 字段可以替代这些样板。

## 4. Hook 级 gating

**点子**：claude-code 的按 agent 的 `hooks` 字段 —— 注册作用域限定在某一个 agent profile（而非全局）的 PreToolUse / PostToolUse 处理器。

**提出的理由**：如今我们的 hooks（`openprogram/plugins/hooks.py`）会派发给所有已加载的插件。没法说“这个 agent 跑这个 hook，那个 agent 不跑”。

**我们没做的理由**：
- 我们有 hook 派发，但对它的使用有限 —— 只接了 `chat.before_send` 和 `tool.before_use`。在全局机制尚未被充分使用之前就加按 agent 的作用域，为时过早。
- claude-code 的按 agent hooks 确实存在，但它的文档暗示大多数用户是通过插件（host 级）注册 hook 的，而非按 agent。
- 统一 gating 模型（本文件夹）已经覆盖了**哪些 tools/skills/MCP** 这个问题。Hooks 是一个**调用如何通过**的问题 —— 是正交的。加上按 agent 的 hooks 并不改变 gating 模型，只是多加一个旋钮。

**何时重新考虑**：当有人写出一个需要按 agent 表现不同的插件时。大概率还要几个月。

---

## 5. 单一 ruleset 迁移（opencode 风格）

**点子**：用单一的 `permission: [{pattern, action}]` 列表替换三个块（`tools`、`skills`、`mcp`）。

**提出的理由**：更干净、单一事实来源、以后更容易加入第 4 种扩展类型。

**我们没做的理由**：
- 按类型分的字段形态本身已经是自文档化的。迁移将是纯重构，除了语法层面之外不带来任何新能力。
- 向后兼容成本 —— 用户 `~/.openprogram/agents/` 里现存的每个 agent profile 都得有迁移代码或一个弃用周期。
- Opencode 的 ruleset 是**首次匹配获胜**的，这意味着规则顺序很重要。按类型分的结构没有顺序方面的顾虑（每个块都是独立的），少一个会自伤的坑。

**何时重新考虑**：当我们加到第 5 或第 6 种受 gating 的扩展类型、schema 开始显得臃肿时。如今我们有 3 种，这是它停止物有所值之前的上限。

---

## 如何重新考虑其中任何一项

1. 找到症状 —— 一条用户抱怨、一次真实事故，或一个带具体用例的功能请求。
2. 与本文档交叉比对，看看当初的推理是什么。
3. 如果当初的推理已被推翻（例如“没有参考实现有这个”已不再成立，因为 opencode 上个月加了它；或“没有用户痛点”已不再成立，因为我们为此收到了 5 个 issue），则推进实现，并更新本文档以记录新的背景。
4. 如果当初的推理仍然成立，就用这里的分析回绝该请求。
